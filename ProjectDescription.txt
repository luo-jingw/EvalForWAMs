LingBot-VA × ViDiT-Q 量化项目 — 高层交接说明

  项目本质

  对 LingBot-VA（5.09B params bf16 WAN transformer）做 ViDiT-Q 量化（W8A8
  baseline + 论文同名 ViDiT 方法 = SmoothQuant + QuaRoT + static activation），
  在 RoboTwin 15 任务上评估 success rate / GPU 显存峰值 / 推理时延，做 bf16 vs
  W8A8 dynamic vs W8A8 viditq(smooth+quarot+static) 三方对比。评估性研究项目，
  要求忠实复刻 ViDiT-Q 方法，不私自简化。

  关键约定

  1. 量化方法：per-channel asymmetric weight（int8 + int16 zero-point）。
     激活量化两条路：
     - **dynamic**（Phase 26a-1）：act_quant_bf16_with_sum 每行算 amax；不需 calib
     - **static**（Phase 33）：act_quant_bf16_with_sum_static 用 calib 推导的标量 scale；
       per-tensor static = max(act_absmax) / 127（如果 smooth 开启，则在 post-smooth
       absmax 上算；rotation 不补偿因为 orthogonal 不放大 max）
  2. SmoothQuant 方向陷阱（Phase 36 bug history）：upstream channel_mask 公式 w^α/a^(1-α)
     applied as W/mask + x*mask；canonical 形式 s = 1/channel_mask = a^(1-α)/w^α applied
     as W*s + x/s。两者数学等价但方向相反，错用方向会**放大 outlier 而非压制**（plumbing
     test 都过因为 unquantized round-trip 不变；量化后才暴露）。我们用 canonical，buffer
     名 act_channel_div 反映其作为 divisor 的语义
  3. QuaRoT 不物化 H 矩阵（per-Linear in_features^2 bf16 矩阵在 14336 宽度下 ≈410 MB
     × 30 down_proj = 12 GB，不可接受）。用 structured Hadamard：per-Linear int8 sign
     vector + butterfly + had12/had28 head。WAN dim 3072 = 12 × 2^8，14336 = 28 × 2^9
  4. CUDA kernel 来源：整文件 verbatim 拷贝 ViDiT-Q 的 w8a8_gemm_cuda.cu，仅做一处扩展：
     加 typename OutT 模板参数 + DtypeTraits<__nv_bfloat16> 特化，让 ViDiT-Q 原本只支持
     fp16 输出变成 fp16 + bf16 双支持。act_quant_bf16.cu 是 LingBot-VA 自己加的（因为
     ViDiT-Q upstream 的 quant kernel hidden ≤ 8192，下不去 14336 down_proj）
  5. fp16 launcher 保留：作为对照实验，bench 时与 bf16 数值对比
  6. 第三方仓库工作树 == upstream（lingbot-va/.git_orig HEAD）。所有研究代码
     在 PTQEval/。修改 upstream 是反模式；新功能 fork 进 PTQEval
  7. 任务集：
     - SELECTED_15_TASKS：生产 eval（1 long + 4 medium-long + 10 short）× 25 episode
     - CALIB_TASKS_ALL：50 task × 5 episode，给 calib 录视频；derive_calib_ptq 默认按
       task_list 过滤回 15 task 子集做 replay（calib 跟 eval 同分布）
  8. RoboTwin task_config：demo_clean（背景固定）vs demo_randomized（背景/光照/桌高
     随机化）。production eval 跑 demo_randomized
  9. P5 commit 风格：每 milestone 单句 commit message（参考 `git log --oneline`）
 10. **NO _v2 / _new / 向后兼容路径**：不要 `foo_v2.py / results/calib_data_v2/ /
     runtime_args_X_new.yaml` 这类后缀化命名。直接复用原路径、覆盖、commit；git
     是 history。例外仅在用户明确要求并行保留旧产物时
 11. **3 个 eval-tier peer 脚本零交叉调用**：run_eval.py（eval orchestrator）、
     collect_calib_videos.py（calib step 1）、derive_calib_ptq.py（calib step 2）
     不互相 subprocess wrap；共享逻辑抽到 _pool_runner.py 私有模块共同 import

  Phase 39 (3-way cross_summary, demo_randomized, 终版结论):

  | variant | Mean SR | Δ SR vs bf16 | Mean ms | Speedup |
  |---|---|---|---|---|
  | bf16                          | 0.871 | —        | 4657 | 1.00× |
  | viditq_w8a8_dynamic (SQ+QR)   | 0.869 | -0.2 pp  | 3406 | **1.37×** |
  | viditq_w8a8_static (SQ+QR+ST) | 0.584 | -28.7 pp | 3575 | 1.30× |

  - Paper-faithful "ViDiT-Q W8A8" 复现 = **dynamic 行**：1.37× 加速 + SR 几乎持平
  - **SR 损失 100% 归因于 static activation quant**：dynamic（去掉 static）SR 跟 bf16
    几乎一致（-0.2pp），加上 per-tensor static scale 才崩 -28.7pp。Long-horizon
    task 是重灾区：put_bottles_dustbin 0.80→0.00、handover_block 0.96→0.08、
    beat_block_hammer 1.00→0.36。可能原因：per-tensor 标量对 activation
    分布波动过敏，clipping outlier 同时把信号也压掉
  - dynamic 速度反而比 static 略快 169 ms（5%）：fused dynamic amax kernel 跟
    static skip-amax 在 launch jitter 面前差不多，static 还多一次 scale_x[N]
    broadcast
  - calib_data 用 demo_randomized 视频（371 ep → per_task_ep=5 共 75 ep）做 derive；
    SR 跟之前 demo_clean calib 几乎相同（0.587→0.584）说明分布 mismatch 不是 SR
    退化主因，是 static 方法本身的 limitation

  - run_eval.py / collect_calib_videos.py 都通过 _pool_runner.kill_session 清理子进程，
    用 /proc tree walk 抓 torch.distributed.run 用 setsid 跑掉的 worker
  - calib hook 走 fcntl.flock + per-PID tmp filename，多 worker pool 并发写
    calib_data.pth 安全
  - **derive_calib_ptq 重大教训（避免重蹈）**：
    旧版（单 GPU）在 parent process 里 in-process load VA_Server（占 ~30 GB GPU
    显存），最后再 subprocess ptq.py 想加载第二份模型 → CUDA OOM。新版（commit
    30b032a）orchestrator 是纯 stdlib（不 import torch），worker 是单独 subprocess，
    worker 退出后 CUDA context 完全释放，再 spawn ptq subprocess 不冲突
  - **derive_calib_ptq replay 策略**：obs_data_*.pt 是 server 在 `_compute_kv_cache`
    时存的 4-frame chunk（VAE state 已 primed）。fresh VAE state 喂 4-frame 会 shape
    错（AutoencoderKLWan.avg_shortcut 期望 post-downsample 2 frames）。replay 用
    reset + 1-frame `_infer` per chunk，跟 eval_client 第一次 infer 时发 first_obs
    （单 dict 不是 list）相符。1-frame 牺牲 chunk 内时间多样性，但 diffusion loop
    本身吃 ~20 transformer forward，75 ep × ~10 chunk = 750 calls 已经够 per-channel
    max 收敛
  - **compute_breakdown chart quirk**：transformer_ms + action_head_ms 有时和 mean_total_ms
    不一致（前者大于后者 ~5%）。原因是 PerfProbe stage timer 在 CFG batching / dual-pass
    （video + action）时存在 overlap。chart 用 `other = max(0, total - trans - head)`
    导致 other 段 clamp 到 0；total 标签放在 bar 实际末端（sum-of-segments）而非 total_ms
    位置，避免 label 撞段内字

  - --use_fast_math 在 setup.py 里开着（faithful to ViDiT-Q upstream）。引入 ~1 fp32
