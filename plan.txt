LingBot-VA on RoboTwin: Success Rate Eval + Memory & Latency Monitoring Plan
===============================================================================

0 PRINCIPLES  (apply to every Phase below)
-------------------------------------------------------------------------------
P1 Research-oriented project. Faithful reproduction of published methods
   takes precedence over implementation convenience.
P2 No private simplification or optimization of an algorithm. Equivalent
   engineering means are permitted (e.g., C++ template instantiation,
   verbatim kernel transcription, type-aware refactor without semantic
   change).
P3 Copy + rename + modify as the default extension pattern. Avoid
   modifying third-party repositories in place; favor a fork inside our
   own tree.
P4 First-principles, straight-through first. Abstract only when a second
   concrete instance demands it. Premature abstraction over a single
   instance is forbidden.


1 PROBLEM DEFINITION
-------------------------------------------------------------------------------
1.1 Current state
- LingBot-VA inference server and RoboTwin client are wired through localhost
  port. Server: lingbot-va/wan_va/wan_va_server.py. Client:
  lingbot-va/evaluation/robotwin/eval_polict_client_openpi.py.
- Server emits actions only. No memory probe. No latency probe.
- RoboTwin client writes per-task success rate to ${save_root}/<task>/.
- Conda envs: lingbot-jw (server) and RoboTwin-jw (client).

  /home/arash/EvalForWAMs/models/lingbot-va-posttrain-robotwin.

1.2 Problem

- No per-stage inference latency record.
- No unified runner for smoke / single-GPU / multi-GPU modes.
- No script to merge SR with perf metrics into one summary.
- No live status dashboard during multi-task eval.

1.3 Goal
- Add explicit perf probe inside server. Five stages: init, text_encoder,
  vae_encode, transformer, action_head.
- Per inference call: peak allocated memory and elapsed milliseconds per
  stage, written as one JSONL line.
- One launcher script with three modes: smoke, single, pool.
  pool replaces the original multi mode and uses worker-pool + queue.
- One aggregator script: reads perf JSONL + RoboTwin SR. Writes summary.csv
  and summary.json.
- Default output root: /home/arash/EvalForWAMs/results/<variant_tag>
  (variant_tag = bf16 for the baseline checkpoint).
- Live status dashboard for watch(1).
- All modified files keep a .bak copy beside them.
- Selected 15 tasks, test_num=25 per task. Task list encoded as
  SELECTED_TASKS in run_eval.sh.
- Resume-aware: per-task res.json with total_num < test_num triggers re-run;
  done tasks are skipped unless --rerun_all.
- VRAM-aware: each worker waits until target GPU has >= MIN_FREE_MB free.


2 SYSTEM STRUCTURE
-------------------------------------------------------------------------------
2.1 Modules
M1 perf_probe          Owns per-stage timing and memory probes.
M2 instrumented_server Inserts probe calls around stages in wan_va_server.py.
M3 perf_log            One JSONL file per server process. Probe owns writes.
M4 robotwin_sr_reader  Reads RoboTwin SR output files.
M5 aggregator          Merges perf_log and SR into summary files.
M6 orchestrator        Single shell entry. Dispatches smoke / single / pool.
                       pool mode owns a shared queue file and a flock-based
                       pop primitive; each worker locks one GPU.
M7 monitor             Read-only dashboard for watch(1). Scans latest log
                       dir under save_root/logs/ and the perf log dir.

2.2 State ownership
- Per-stage timing and memory: owned by M1. Other modules read only.
- Per-episode SR: owned by RoboTwin client (existing). M5 reads only.
- Summary tables: owned by M5. Written once per eval run.
- Process and GPU assignment: owned by M6.
- Shared task queue file: owned by M6 pool mode. flock serializes pop.
- pidfile per server log: owned by M6 start_server / kill_pid_file.


3 INTERFACE DESIGN
-------------------------------------------------------------------------------
3.1 M1 perf_probe (Python)
File: lingbot-va/wan_va/utils/perf_probe.py

  @dataclass
  class StageRecord:
      stage: str
      elapsed_ms: float
      peak_alloc_mb: float
      peak_reserved_mb: float

  @dataclass
  class CallRecord:
      call_id: int
      task_name: str
      timestamp: str
      stages: list[StageRecord]
      total_ms: float

  class PerfProbe:
      def __init__(self, log_path: str, task_name: str,
                   device: int) -> None: ...
      def begin_call(self) -> None: ...
      def stage(self, name: str) -> "StageContext": ...
      def end_call(self) -> CallRecord: ...
      def close(self) -> None: ...

  class StageContext:
      def __enter__(self) -> "StageContext": ...
      def __exit__(self, exc_type, exc, tb) -> None: ...

Behavior
- begin_call: reset_peak_memory_stats(device).
- stage(name): on enter: torch.cuda.synchronize(); start CUDA Event;
  reset_peak_memory_stats(device). On exit: end CUDA Event;
  synchronize; read max_memory_allocated / max_memory_reserved;
  append StageRecord.
- end_call: aggregate StageRecord list; compute total_ms; write one JSONL
  line to log_path; return CallRecord.

3.2 M2 instrumented_server
File: lingbot-va/wan_va/wan_va_server.py (modified, backup as .bak)

3.4 M5 aggregator (Python)
File: lingbot-va/evaluation/robotwin/calc_perf_stat.py

  def main(save_root: str, perf_log_dir: str, out_dir: str) -> None: ...

Per-task perf record selection
- Walks perf_log_dir for files matching
  <task_name>_rank<N>_<YYYYMMDD_HHMMSS>.jsonl.
- task_name comes from the filename, not from per-record fields.
- For each task, only the file with the largest mtime is consumed;
  older files (from interrupted prior runs) are skipped and the skip
  list is printed. Files remain on disk.

Outputs
- ${out_dir}/summary.csv columns: task_name, num_episodes, success_rate,
  init_peak_mb, mean_total_ms, p50_total_ms, p95_total_ms,
  mean_transformer_ms, mean_action_head_ms, peak_alloc_mb,
  peak_reserved_mb.
- ${out_dir}/summary.json with same data plus per-stage breakdown
  (per_stage_mean_ms, per_stage_peak_alloc_mb, per_stage_peak_reserved_mb).

3.5 M6 orchestrator (Bash)
File: lingbot-va/evaluation/robotwin/run_eval.sh

CLI
  run_eval.sh --mode {smoke,single,pool}
              [--task_name <name>]        # smoke, single
              [--test_num <int>]          # default 25, smoke uses 1
              [--save_root <path>]        # default
                                          # /home/arash/EvalForWAMs/results/bf16
              [--perf_log_dir <path>]     # default ${save_root}/perf
              [--model_path <dir>]        # default
                                          # models/lingbot-va-posttrain-robotwin
              [--gpu_id <int>]            # default 0 for smoke, single
              [--seed <int>]              # default 0
              [--min_free_mb <int>]       # default 40000
              [--gpu_wait_timeout <int>]  # default 0 (wait forever)
              [--gpu_poll_interval <int>] # default 30
              [--rerun_all]               # pool: ignore prior res.json

Behavior
- smoke: 1 GPU, 1 task (default adjust_bottle), test_num=1.
  Starts 1 server, waits for port, runs 1 client, kills server.
- single: 1 GPU, sequential loop. If --task_name set, runs that one;
  otherwise loops over SELECTED_TASKS. New server per task.
- pool: queue = tasks needing run (per task_needs_run). Picks usable
  GPUs (free VRAM >= MIN_FREE_MB) at start; spawns one worker per usable
  GPU. Each worker pops next task from the shared queue (flock), starts
  a fresh server on its GPU, runs the client, kills the server, loops.
  Exits when queue empty.

4 RUNTIME FLOW
-------------------------------------------------------------------------------
4.1 Server lifecycle (per process)
- Read CLI. If --perf_log_dir set: construct PerfProbe with
  log_path=${perf_log_dir}/${perf_task_name}_rank<N>_<ts>.jsonl.
- probe.begin_call(); with probe.stage("init"): load vae, tokenizer,
  text_encoder, transformer, optional vae_half. probe.end_call().
- Open websocket via run_async_server_mode.
- Per incoming obs: probe.begin_call(); branch on flags:
    reset            : stage(text_encoder) inside _reset.
    compute_kv_cache : stage(vae_encode) inside _encode_obs;
                       stage(transformer); stage(action_head).
    infer            : stage(vae_encode) inside _encode_obs;
                       stage(transformer); stage(action_head).
  probe.end_call() in finally.

4.2 Client lifecycle (per task)
- RoboTwin client unchanged except for the robowin_root path fix.
- Per-episode metrics written to
  ${save_root}/stseed-<seed>/metrics/<task_name>/res.json.

4.3 Aggregation flow
- After all clients exit, run calc_perf_stat.py with save_root,
  perf_log_dir, out_dir=${save_root}/summary.
- M5 walks perf_log_dir for *.jsonl; for each task keeps newest by mtime.
- M5 walks save_root/stseed-*/metrics/*/res.json for SR.
- Joins on task_name. Writes summary.csv and summary.json.

4.4 Pool worker flow (per worker)
- Loop:
    t = pop_task(queue_file, lock_file)
    if t empty: break
    wait_for_gpu_memory(gpu)
    start_server(gpu, port, mport, t, server_log)
    wait_for_port(port, 600)
    start_client(t, port, test_num, client_log, gpu)
    kill_pid_file(server_log.pid)

4.5 Backup flow
- Before any in-place edit on existing tracked files, the modifying step
  creates <file>.bak if absent. No automated restore. Manual revert via cp.

PART II  Post-Training Quantization of LingBot-VA via ViDiT-Q
===============================================================================

8 PROBLEM DEFINITION
-------------------------------------------------------------------------------
8.1 Current state
- Baseline bf16 eval at results/bf16/summary/summary.csv: 15 tasks,
  test_num=25 per task, mean SR ~0.86.
- Transformer parameters: 5.09 B, all 2D nn.Linear weights.
- Memory: init peak 23 301 MB; runtime peak ~32 000 MB.
- Inference dtype: bf16 throughout (param_dtype = torch.bfloat16).
- Distributed wrap: _configure_model -> shard_model (FSDP) at server init.
- ViDiT-Q source tree at /home/arash/EvalForWAMs/ViDiT-Q/. Standalone
  package quant_utils/qdiff/ is reusable as-is.

8.2 Problem
- No quantized variant exists.
- No PTQ pipeline for WanModel.
- No calibration data collection for LingBot-VA's two-stream
  (video_mode and action_mode) Transformer.
- No cross-checkpoint comparison (bf16 vs W8A8 vs W4A8).
- Server load path is hardcoded to from_pretrained; no quantized branch.
- No CUDA kernel for W8A8 GEMM matching WanTransformerBlock shape.

8.3 Goal
- Vendor qdiff package into lingbot-jw env as editable install.
- Provide QuantWanTransformer subclass with PTQ surgery on nn.Linear.
- Provide calibration runner that exercises both action_mode values.
- Provide PTQ runner that writes quant_params.pth.
- Server gains optional quantized load path; bf16 path unchanged.
- Provide W8A8 and W4A8-mixed-precision configs.
- Custom CUDA kernel for W8A8 Linear matching WanTransformerBlock,
  enabling actual weight memory reduction at inference.
- Cross-checkpoint comparison output extends summary aggregator.

8.4 Non-goals
- No retraining. No QAT. Only post-training quantization.
- No attention-matmul quantization. Only nn.Linear weight + activation.
- No quantization of text_encoder or VAE. Only transformer.

8.5 Pivot (after Phase 13)
Phase 8-13 produced a working algorithm-simulation path (qdiff-based
QuantizedLinear that holds bf16 dequant weights and computes dynamic
activation quant in pure Python). Smoke verified SR preservation but
showed runtime memory and latency overheads with no end-user payoff:
the bf16 dequant weight has the same footprint as FP, and DynamicQuantizer
intermediates inflate activation peak by ~30%.

We pivot to a kernel-only path:
- True INT8 (and packed INT4) weight storage, BF16-native CUDA GEMM.
- Per-channel symmetric weight scale, per-token symmetric activation scale.
- Algorithm-simulation code is removed (Phase 14). The eval is meaningful
  only against the kernel checkpoint.
- Both W8A8 and W4A8 variants are implemented.


9 SYSTEM STRUCTURE  (kernel-first, Phase 14+)
-------------------------------------------------------------------------------
9.1 Modules
Q1 quant_configs          YAML schema per variant: target Linear regex,
                          weight bitwidth, runtime args. Stored under
                          quant/viditq/configs/.
Q2 ptq_kernel             CLI script that walks an FP WanTransformer3DModel
                          and produces a state_dict with int weights +
                          per-channel scales for every target Linear.
                          Single output file int_weights_<variant>.pth.
                          No dependency on qdiff.

                          quant + W8A8 GEMM that returns BF16. Source
                          under quant/viditq/kernel/csrc/.

                          weight is INT4 (packed 2 per INT8 byte).
Q5 quant_linear_modules   Python nn.Module wrappers
                          QuantWanLinearW8A8 / QuantWanLinearW4A8 around
                          Q3 / Q4 kernels. state_dict-load compatible
                          with Q2 output.
Q6 quant_block_kernel     QuantWanTransformerBlockWithCudaKernel that
                          mirrors WanTransformerBlock but swaps the 6
                          target Linears (4 attn proj + 2 ffn) with Q5
                          modules. Norm / rope / attention matmul stay FP.
Q7 quant_server_loader    --variant / --variant_args dispatch already in
                          wan_va_server.py. Resolves importlib path and
                          calls Q8.load_quant_model. No change.
Q8 viditq_loader          load_quant_model entry. Loads FP model, swaps
                          every block for Q6, loads Q2 state_dict, returns
                          model.
Q9 cross_ckpt_aggregator  Reads multiple save_roots and emits
                          cross_summary.csv joining bf16 + W8A8 kernel +
                          W4A8 kernel.

9.2 State ownership
- INT weights + per-channel scales: produced by Q2, persisted to
  int_weights_<variant>.pth. Loaded by Q8.
- Active checkpoint selection: owned by Q7 via --variant + --variant_args.
- Per-token activation scale (runtime buffer): owned by Q5 module instance.
- Cross-checkpoint table: owned by Q9.

10.4 Q4 kernel_w4a8 (CUDA)

11 RUNTIME FLOW  (kernel-first)
-------------------------------------------------------------------------------
11.1 PTQ flow (offline, per variant)
- Q2 loads FP WanTransformer3DModel on a single GPU or CPU.
- For each nn.Linear whose name does NOT match remain_fp_regex:
    scale_w = max(|W|, dim=in_features) / qmax   # qmax = 127 for w8, 7 for w4
    int_W   = round(W / scale_w).clamp(-qmax-1, qmax).to(int_dtype)
    if w4: pack 2 nibbles per int8 byte (little-endian)
    record (int_W, scale_w[, bias])
- Dump dict[full_name -> IntLayerEntry] to int_weights_<variant>.pth.

11.2 Quantized inference flow (server)
- Server starts with --variant viditq --variant_args <yaml>.
- Q7 branch imports wan_va.quant.viditq.loader and calls Q8.load_quant_model.
- Q8:
    1. Load FP WanTransformer3DModel (cheap source of non-quantized graph).
    2. For each block, instantiate Q6 with the appropriate
       QuantWanLinear[W8A8|W4A8] class derived from layer_config.weight_bits.
    3. Load int_weights state_dict into the new blocks' Linear modules.
    4. Drop the original FP blocks; keep everything else (embeddings,
       norms, condition_embedders) as FP.
- Returned module is assigned to self.transformer. FSDP wrap is skipped
  (single-GPU eval; quantized modules are not FSDP-compatible).
- Per-stage probe (M2) unchanged: stage transformer and stage action_head
  now measure kernel-backed forward.

11.3 Kernel forward sequence (per target Linear)
  x_bf16  -> [Q3.act_quant_bf16] -> (x_int8, scale_x_bf16)
                                  -> [Q3.w8a8_gemm_bf16 or Q4.w4a8_gemm_bf16]
                                  -> y_bf16
- Activation quant is per-token symmetric (scale computed online per row).
- Weight quant is per-channel symmetric (scale frozen at PTQ).

11.4 Cross-checkpoint comparison flow
- Q9 reads multiple save_roots, one per variant.
- Each save_root has its own perf/ and stseed-*/metrics/.
- Output: cross_summary.csv joined on task_name.

15 EXTENSIBILITY  (adding a new quantization method)
-------------------------------------------------------------------------------
NOTE: After Part IV (PTQEval migration, Phases M1-M8), this section's
directory convention is SUPERSEDED. Adding a new method or new WAM now
follows Section 24 (PTQEval system structure). The loader contract below
remains valid; the only change is the resolution path:
    OLD: wan_va.quant.<name>.loader              (in lingbot-va tree)
    NEW: ptqeval.wam.<wam>.method.<name>.loader  (in PTQEval tree)
Section 26 Phase M7 added this note; PTQEval/README.md mirrors Section 24.

15.1 Loader protocol
The single contract that the server depends on:

  # lingbot-va/wan_va/quant/<name>/loader.py
  def load_quant_model(
      wan_model_path: str,
      variant_args: dict,
      device: torch.device,
      dtype: torch.dtype,
  ) -> nn.Module: ...

Returns
- A ready-to-use nn.Module that is signature-compatible with the
  original WanModel transformer (same forward(*args, **kwargs) as in
  lingbot-va/wan_va/modules/model.py::WanModel). The server assigns it
  to self.transformer; no other code path is aware of the variant.

Notes
- variant_args is the OmegaConf-loaded dict from --variant_args yaml.
  Schema is method-internal and may include checkpoint paths, runtime
  flags such as use_kernel, mixed-precision bitwidth lists, etc.
- The loader is responsible for everything: instantiating the wrapper
  class, loading FP weights, loading quant params, optional kernel
  swap. The server does not import method-internal classes.

15.2 Directory convention
Every variant lives under exactly:
  lingbot-va/wan_va/quant/<name>/
A variant directory MUST contain:
  __init__.py
  loader.py                   (exposes load_quant_model)
A variant directory MAY contain:
  ptq.py                      standalone CLI: FP -> int weights + scales
  calib_*.py                  optional calibration if the method needs it
  configs/*.yaml              method-specific layer configs and runtime args
  kernel/                     CUDA extension; build under setup.py here
  block/                      block-level kernel-replacement modules
Rule: all method-specific code stays inside this directory. The eval
pipeline (M1-M7, Q9 cross_ckpt) is method-agnostic.

15.3 Add-a-method checklist
Step 1  Create lingbot-va/wan_va/quant/<name>/loader.py with
        load_quant_model.
Step 2  Implement whatever offline pipeline the method needs (calib,
        PTQ, kernel build). Outputs go under
        results/<variant_tag>/calib/.
Step 3  Run eval:
            bash run_eval.sh --mode pool --variant <name>
                --variant_args lingbot-va/wan_va/quant/<name>/
                               configs/runtime_args_<tag>.yaml
        Default save_root becomes results/<variant_tag>/.
Step 4  Add the new save_root to the cross_ckpt_aggregator input set.

15.4 Out of scope
- The eval pipeline (M1-M7, Q10) does not change when a new variant is
  added.
- No registry, no factory, no plugin manager. Resolution is exactly
  importlib.import_module(f"wan_va.quant.{variant}.loader").

16 PROBLEM DEFINITION
-------------------------------------------------------------------------------
16.1 Current state (after Phase 20)
- Loader (Section 10.8) loads FP transformer, wraps every block with
  QuantWanTransformerBlockWithCudaKernel, loads int_weights state_dict,
  returns nn.Module. Server smoke verified.
- Existing PTQ (Phase 18) writes per-channel SYMMETRIC weight quant:
  state_dict = {<name>.int_weight, <name>.scale_weight, <name>.bias}.
- Existing GEMM kernel (Phase 15) is a scalar 1-thread-per-output kernel.
  Numerically correct, but ~25x slower than bf16 baseline at WAN shapes.
- Phase 21 run confirmed kernel slowness blocks full eval.

16.2 Problem
- Current quant method (symmetric weight) is NOT the ViDiT-Q W8A8 method.
  Faithful evaluation requires per-channel asymmetric weight with int16
  zero-point per output channel.
- Current kernel does not use Ampere INT8 tensor cores (mma.sync.aligned.
  m16n8k32.s32.s8.s8.s32). Wall-clock unusable.
- The asymmetric weight method requires an additional per-row activation
  sum (sum_A) in the epilogue. Current act_quant_bf16 does not produce it.

16.3 Goal
- Replace ptq.py with asymmetric per-channel weight quant matching
  ViDiT-Q's W8A8 / W4A8 configs (weight.sym=False, act.sym=True).
- Port ViDiT-Q's MMA-based GEMM kernels (w8a8_gemm_cuda.cu and
  w4a8_per_channel_gemm_cuda_qserve.cu) verbatim into our tree.
- Extend act_quant_bf16 to also output sum_x (per-row dequant sum).
- Add bf16 output as a new OutT template instantiation alongside
  ViDiT-Q's existing fp16 + int32 paths. fp16 path preserved as
  numerical reference for evaluation.
- All ViDiT-Q template branches preserved: weight_asym (true/false),
  has_bias (true/false), OutputDtype (fp16/bf16/int32), ScaleMulMode
  (kMode1/kMode2), K_STAGE, CTA/WARP tile shapes.

16.4 Faithfulness constraint
- Quant method must match ViDiT-Q exactly. No private simplification.
- Kernel must match ViDiT-Q structurally. Permitted additions: a new
  OutT template parameter that selects bf16 epilogue; corresponding
  DtypeTraits<__nv_bfloat16> specialization. No deletion of existing
  template branches, ScaleMulMode variants, OutputDtype variants.
- Reasoning: this work is the evaluation harness for ViDiT-Q on
  LingBot-VA. The published method must be reproduced as-is to keep
  numerical and design comparability.

16.5 Non-goals
- No new quantization method invented here. Only ViDiT-Q W8A8 + W4A8.
- No fusion with norm or activation (ViDiT-Q kernel structure is
  bare GEMM + epilogue; norm stays as separate FP kernel).
- No multi-GPU GEMM split.

18.2 Q2 ptq_kernel (rewritten)
File: lingbot-va/wan_va/quant/viditq/ptq.py

  @dataclass
  class IntLayerEntry:
      int_weight: torch.Tensor      # int8 [C_out, C_in] or [C_out, C_in/2]
      scale_weight: torch.Tensor    # bf16 [C_out]
      zp_weight: torch.Tensor       # int16 [C_out]
      bias: Optional[torch.Tensor]  # bf16 [C_out] or None

  def compute_int_state_dict(
      model: nn.Module,
      remain_fp_regex: str,
      weight_bits: int,             # 8 or 4
      weight_sym: bool,             # always False for Part III
  ) -> dict[str, IntLayerEntry]: ...

Asymmetric per-channel formula (matches qdiff StaticQuantizer asym path):
  for each Linear, with weight W[C_out, C_in]:
      x_max = W.max(dim=1)                          # [C_out]
      x_min = W.min(dim=1)                          # [C_out]
      n_levels = 2**weight_bits
      delta = (x_max - x_min) / (n_levels - 1)      # scale, [C_out], fp32
      zp_f  = round(x_min / delta) + (n_levels / 2) # [C_out], fp32
      int_w = round(W / delta + zp_f)               # [C_out, C_in]
      int_w = clamp(int_w - n_levels/2, -n_levels/2-1, n_levels/2)
              .to(int8)                             # signed int8
      # Notes:
      #   delta is stored as bf16 in scale_weight.
      #   zp_f - n_levels/2 is stored as int16 in zp_weight to match
      #   the kernel's short2 load.
For W4 case (n_levels = 16), additionally pack 2 nibbles per byte
  (low nibble = col 2c, high nibble = col 2c+1, signed).

Flat state_dict keys:
  <name>.int_weight     int8 [C_out, C_in] or [C_out, C_in/2]
  <name>.scale_weight   bf16 [C_out]
  <name>.zp_weight      int16 [C_out]
  <name>.bias           bf16 [C_out]   (omitted if absent)

18.5 Q11 epilogue_dtype_traits

  template <typename T> struct DtypeTraits;

  template <> struct DtypeTraits<__half> {
      using packed2 = __half2;
      __device__ static float  to_float(__half x);
      __device__ static float2 to_float2(__half2 x);
      __device__ static __half2 from_float2(float2 x);
  };

  template <> struct DtypeTraits<__nv_bfloat16> {
      using packed2 = __nv_bfloat162;
      __device__ static float  to_float(__nv_bfloat16 x);
      __device__ static float2 to_float2(__nv_bfloat162 x);
      __device__ static __nv_bfloat162 from_float2(float2 x);
  };

The MMA kernel uses Trait::to_float / to_float2 to convert scale + bias
to fp32, and Trait::from_float2 to write packed2 results into RC.

19 RUNTIME FLOW
-------------------------------------------------------------------------------
19.1 PTQ flow (offline, per variant)
- Q2 loads FP WanTransformer3DModel.
- For each nn.Linear whose name does NOT match remain_fp_regex:
    apply asymmetric per-channel formula (Section 18.2).
    For W4: pack 2 nibbles per byte after the int_w computation.
- Dump dict[full_name -> IntLayerEntry] to int_weights_<variant>.pth.

19.2 Quantized inference flow (server)
- Server starts with --variant viditq --variant_args <yaml>.
- Q7 imports wan_va.quant.viditq.loader and calls Q8.load_quant_model.
- Q8:
    1. Load FP WanTransformer3DModel.
    2. For each block, instantiate Q6 with the appropriate QuantWanLinear
       class. Constructor calls from_fp_linear which initializes int /
       scale / zp / bias buffers from FP weights using the same
       asymmetric formula (matches Q2 numerically).
    3. Load int_weights_<variant>.pth into the kernel modules. The
       loaded values overwrite the from_fp_linear initialization.
- Returned module assigned to self.transformer. FSDP wrap skipped.

19.3 Kernel forward sequence (per target Linear)
  x_bf16  -> [Q10.act_quant_bf16_with_sum] -> (x_int8, scale_x, sum_x)
                                            -> [Q3.w8a8_of_bf16_bias_weight_asym
                                                or Q4.w4a8_of_bf16_bias_weight_asym]
                                            -> y_bf16
Inside kernel epilogue (per ViDiT-Q):
  acc_int32 -> psums_fp32 = acc * scale_A * scale_B
  if weight_asym: psums_fp32 += sum_A * zp_B * scale_B
  if has_bias:    psums_fp32 += bias
  output_bf16   = DtypeTraits<__nv_bfloat16>::from_float2(psums_fp32)

19.4 Cross-checkpoint comparison flow
Unchanged (Section 11.4).
