10 INTERFACE CONTRACTS
-------------------------------------------------------------------------------
10.1 yaml schema
     remain_fp_regex: <regex>           # outer Linears kept FP
     weight: { n_bits: 8|4, sym: false }
     act:    { n_bits: 8,   sym: true  }
     optional: smooth_quant_alpha, quarot, static_act, bit_alloc

10.2 PTQ asym formula (per Linear with weight W[C_out, C_in])
     x_max = W.amax(1).clamp_min(0); x_min = W.amin(1).clamp_max(0)
     n_levels = 2**weight_bits
     delta = (x_max - x_min) / (n_levels - 1)
     zp    = round(x_min / delta) + n_levels/2
     int_w = clamp(round(W/delta) - zp, -n_levels/2, n_levels/2 - 1)
     Storage: delta bf16 in scale_weight; zp int16 in zp_weight.
     W4 case packs 2 nibbles per byte after asym quant (QServe layout).

10.3 Q5 state_dict keys per layer
     int_weight     int8 [C_out, C_in_pack]
     scale_weight   bf16 [C_out]
     zp_weight      int16 [C_out]
     bias           bf16 [C_out]    (optional)
     +PTQ-time optional buffers per Part VI:
       act_channel_div bf16 [C_in]
       quarot_sign     int8 [C_in]
       act_scale_static bf16 [1]
       weight_bits     scalar tensor (mixed-precision dispatch)
       szeros_weight   bf16 [C_out]   (W4A8 QServe epilogue)

10.4 Kernel launchers (after OutT extension)
     w8a8_of{bf16,fp16}_bias_weight_asym(x_int8, w_int8, y, bias,
         scale_x, scale_w, sum_x, zp_w, M, N, K)
     w4a8_of{bf16,fp16}_nobias_weight_asym(...)  (QServe layout)
     Plus weight_sym + no_bias + int32 variants for parity.
     ViDiT-Q convention: M = tokens, N = out_features, K = in_features.

19 KEY CONVENTIONS
-------------------------------------------------------------------------------
19.1 Target Linears (180 total = 30 block x 6)
     per block: attn1.to_q, attn1.to_k, attn1.to_v, attn1.to_out[0],
                ffn.net[0].proj, ffn.net[2]
     cross-attn attn2 stays FP (ViDiT-Q image-DiT convention).
     blocks.0 stays FP (paper remain_fp_regex includes "blocks.0").
     remain_fp_regex semantics: a Linear name matched by the regex is
     left FP and not swapped; the wrapper still iterates over all
     nn.Linear modules but skips matched names.

19.2 SmoothQuant (Phase 36)
     canonical s = a^(1-alpha) / w^alpha
       PTQ:     W' = W * s
       runtime: x' = x / s
     buffer name: act_channel_div (because it divides at runtime).
     ViDiT-Q upstream uses channel_mask = w^alpha / a^(1-alpha) with
     W/mask + x*mask — mathematically equivalent but inverse direction.
     Bug-trap: using upstream formula in canonical orientation amplifies
     outliers (plumbing tests still pass because unquantized round-trip
     is invariant; quantized error blows up only after PTQ).
     alpha = 0.99 for w8a8 (image-DiT default); 0.5665 for w4a8 (paper
     opensora1.2 w4a8_mixed_precision.yaml).

19.3 QuaRoT (Phase 37)
     structured Hadamard (had12 + had28 + butterfly), per-Linear int8
     sign vector + online fast_hadamard_transform (FHT). Do NOT
     materialize H = [C_in, C_in] bf16 (in=14336 -> 410 MB per Linear
     x 30 down_proj = 12 GB). WAN dim 3072 = 12 * 2^8, 14336 = 28 * 2^9.
     FHT runtime CUDA kernel from Dao-AILab/fast-hadamard-transform;
     PTQ-time rotate_weight stays on Python fp64 fallback (FHT lacks
     fp64 path). _HAVE_FHT runtime flag controls the path.

19.4 ViDiT combined (Phase 38)
     PTQ:     W_eff = (W * s) @ H
     runtime: x_eff = (x / s) @ H
     before Phase 24d / 26a-1 quant. State carries both
     act_channel_div and quarot_sign.

19.5 Architecture truth (vs paper)
     ckpt has ONE blocks.0-29 sequence shared by video and action
     streams; all dim=3072. Streams concat along sequence (model.py:
     721-724) and are isolated by FlexAttn mask (model.py:765-772).
     ckpt has 14 outer keys for action (action_embedder, action_proj_out,
     condition_embedder_action) — no action_blocks / action_attn.
     Paper Sec 3.3 "separate backbones same depth d_a<<d_v" is the
     design intent; release ckpt is the simplified token-interleaving
     implementation.
     Consequence for mixed-precision: stream-level bit routing
     (video=W8, action=W4) is impossible (weight shared). Bits are
     allocated by block index only.

Paper-vs-source gap inventory (= scope beyond verbatim port)
  G1 Python binding missing for W4A4. atom.cu is compiled into
     viditq_extension.qgemm (setup.py L154) but kernels/csrc/qgemm/
     gemm_cuda.h declares zero W4A4 launchers and qgemm/pybind.cpp
     registers zero W4A4 m.def entries. ViDiT-Q's release cannot call
     the W4A4 kernel from Python. We add our own decl + m.def +
     Python launcher wrappers.
  G2 Keeper machinery is Atom-only, not ViDiT-Q. atom.cu carries
     KEEPER / KEEPER_* macros, mma_calculate_keeper(), and
     A_keeper / B_keeper params from the Atom paper (Zhao Yilong
     et al. MLSys'24, paper L1872-1874). ViDiT-Q paper 0 mention,
     w4a4_mixed_precision.yaml 0 keeper field, quant_utils 0 keeper
     reference. We strip keeper code paths at port time.
  G3 Bias-fusion missing in atom.cu. dequant() has no bias-load /
     bias-add path. ViDiT-Q's W8A8 kernel already implements
     has_bias=true (w8a8_gemm_cuda.cu); W4A4 omission is LLM-upstream
     lineage. Paper Sec 5.1 discusses fusion of quant + Hadamard into
     LayerNorm/GeLU/residual but does NOT explicitly mention bias-add
     fusion into GEMM; the retrofit matches the W8A8 kernel's
     existing engineering choice. We add has_bias template + register
     FFMA.
  G4 Same bias-fusion gap in csrc/qgemm/w4a8/...qserve.cu (QServe
     LLM upstream). Concurrent retrofit alongside G3.
  G5 Quant scheme yaml-vs-kernel mismatch. w4a4_mixed_precision.yaml
     declares weight.sym=false + act.sym=false (asymmetric), but
     atom.cu compute_gemm_imma + dequant() implement only per-group
     SYMMETRIC INT4xINT4: dequant accumulates `accu += c_frag *
     scale_a * scale_b` with no zp / sum_x / sum_w terms; kernel
     signature has no zp buffer. Adding asym path means a different
     kernel (large rewrite). The mismatch is invisible upstream
     because of G1 — no Python ever calls the kernel, yaml never
     meets kernel. We follow the kernel (per-group symmetric W4A4),
     since the kernel is the only thing that can actually run; treat
     the yaml's sym=false as W4A8-template residue. Documented as
     deviation from yaml-faithful posture.

Quant convention
- Weight (W4A4 layers): per-group SYMMETRIC INT4, group=128 along K
  (kernel-faithful per G5; yaml says asym, kernel only supports sym).
- Activation (W4A4 layers): per-group SYMMETRIC INT4, group=128 along K
  (same kernel constraint).
- Weight (W8A8 layers within W4A4-MP): per-channel asymmetric INT8
  (unchanged from Phase 26/40 pipeline; existing W8A8 wrapper).
- Activation (W8A8 layers within W4A4-MP): per-token symmetric INT8
  (unchanged from Phase 27/40; not strictly yaml-faithful — yaml says
  act.sym=false at top-level — but matches the existing W8A8 kernel
  we are not modifying for this phase).
- Bias: fused into the GEMM epilogue (register-level FFMA inside
  dequant(), one add per output element). The W8A8 kernel already
  implements has_bias=true; W4A4 + W4A8 omit it (LLM upstream
  lineage, see G3/G4). Paper does not explicitly discuss bias-into-
  GEMM fusion; this is engineering carry-over from W8A8 not a
  paper-cited choice. PTQ-time int_weights schema unchanged (bias
  stays on the FP nn.Linear).

QuaRoT (decoded from yaml)
  yaml: quarot.layer_name_regex = ff.net|to_v|to_k|to_q
  Matches 5 of 6 quantizable Linears per block:
    {to_q, to_k, to_v, ffn.net[0].proj, ffn.net[2]}
  attn1.to_out[0] is NOT rotated (input is post-attention head concat,
  already structured). Same structured Hadamard (had12 + had28 +
  butterfly) as Phase 37; dim 3072 = 12*2^8, ffn hidden 14336 = 28*2^9.

SmoothQuant (NOT in W4A4 yaml — paper offers no explanation)
  W4A4 yaml has no `smooth_quant` block and no `viditq` (combined)
  block. The PTQ router quant_utils/qdiff/base/quant_model.py only
  applies SmoothQuant when `smooth_quant` is present in the config,
  so omission is a deliberate yaml choice, not a default fallthrough.
  Paper text does not justify this departure from the Sec 4.2
  static-dynamic combined framework (the closest is Sec D.4
  "without careful tuning"). We follow the yaml (= the published
  W4A4-MP plan) without inventing a rationale.

  NOT reuse W4A8 mixed int_weights.pth (different preprocessing).

Phase 43: PTQ scope trim — skip non-swap-target Linears at quant time
-------------------------------------------------------------------------------
Status:    PLANNED (post-Phase-42; non-blocking for cross_summary).
Trigger:   Phase 42 W4A4 PTQ produced int_weights.pth = 3561 MB / 1189
           keys, of which ~1 GB / ~348 keys are quantized cross-attn
           (attn2) entries that the kernel block.py does NOT swap and
           the loader silently drops via `if k in model_keys` filter.
           Same pattern affects W8A8 / W4A8 ckpts (historical, since
           Phase 26).
Problem:   ptq.py quantizes every nn.Linear that escapes
           layer_config.remain_fp_regex. block._TARGET_SUFFIXES (the
           authoritative list of swap targets: attn1.{to_q,to_k,to_v,
           to_out.0} + ffn.{net.0.proj, net.2}) is enforced ONLY at
           loader/wrapper construction time. cross-attn attn2 + any
           other extra Linears get serialized and then dropped.
Cost (no runtime impact, but real):
  - +~1 GB disk per int_weights.pth (3 ckpts: w8a8 / w4a8 / w4a4 each
    affected; ~3 GB total wasted).
  - +~30 sec PTQ time per ckpt for the redundant _quantize_one calls.
  - +~0.5 sec load time per eval run for the torch.load -> filter.
  - 0 runtime VRAM impact (loader filter drops before any buffer
    materializes on GPU).
Fix (preferred):
  ptq.compute_int_state_dict adds an opt-in `target_only` parameter
  (default False for back-compat). When True, the iterator skips any
  nn.Linear whose suffix is not in block._TARGET_SUFFIXES AND whose
  prefix is "blocks.<i>." (i.e. block-internal Linears beyond the swap
  set). Outer Linears already handled by remain_fp_regex.
  Wire `target_only: true` into all 3 shipped variant yamls (w8a8 /
  w4a8 / w4a4). yaml schema gets one new top-level boolean.
Alternative (less clean):
  Extend remain_fp_regex in each yaml with "attn2" (matches all
  cross-attn). Pro: zero code change. Con: regex semantics drift
  (regex no longer means "kept FP for paper-faithfulness", it now
  also means "skipped because kernel doesn't want it"); brittle if
  block.py ever adds a non-attn2 swap exclusion.
Observational verify:
  Re-run check_ptq_1block.py with target_only=true on the synthetic
  6-Linear "blocks.5.*" container (plus a synthetic cross-attn
  Linear); confirm cross-attn skipped, swap targets unchanged.
  Re-PTQ the 3 production ckpts; expect:
    int_weights.pth size shrinks ~25-30% (W8A8 ~9 GB -> ~7 GB;
    W4A8 ~5 GB -> ~4 GB; W4A4 ~3.5 GB -> ~2.5 GB).
    loader skipped-count drops to ~0.
  Eval SR unchanged (cross-attn was never quantized at runtime
  regardless of ckpt content).
Scope (minimal):
  Single ptq.py edit + 3 yaml edits + check script extension. No
  loader / block / wrapper changes.

44.0 DOUBLE-CHECK FINDINGS (2026-06-29, smoke-verified)
- F1 prompt premise false. obs_data 'task' is a PER-EPISODE randomized
  object-description instruction, not a per-task fixed string. Measured
  unique prompts: results/bf16 365 / 371 ep; results/viditq_w4a8 5240 /
  5000 ep (~1 unique per episode). Episode dirs = <prompt>_<timestamp>,

- F2 disk cache (44a/b/c) scope. precompute-from-a-prior-run prompts do
  not match a fresh demo_randomized eval => mostly miss => swap
  fallback. The disk precompute cache is effective only for reproducible
  prompts (demo_clean, or a seeded replay), NOT production demo_randomized.
- F3 serial residency (44d) is prompt-agnostic: correct peak regardless
  of prompt variety, pays ~+7.3 s per UNIQUE prompt (transformer CPU
  round-trip) = ~+7 s/episode under randomization. Valid deployment
  mechanism; one reset/episode so it is a few % of a multi-minute episode.
- F4 apples-to-apples peak is ALREADY MEASURED. PerfProbe resets peak
  per stage; aggregator stores per_stage_peak_alloc_mb {init,

  working-set peak = max(transformer, action_head, vae_encode). w4a8

- F5 cross-variant inconsistency in the existing local runs: bf16/w8a8
  ran T5 GPU-resident (init 23 GB -> T5 in EVERY stage); w4a8 ran the
  transient swap (init 11.6 GB -> T5 only in the text_encoder stage). So
  F4's per-stage exclusion cleanly fixes w4a8 but NOT bf16/w8a8 (T5 is
  in all their stages). A clean 5-way needs either a consistent re-run

- F6 wiring gap: text_cond_cache (and serve_residency) are read by

44.3 STRUCTURE (modules + state ownership)
- M-TC1 precompute script (offline). Owner of: cache file write.
- M-TC2 text_cond cache (load/store + schema). Single owner of: the
  on-disk cache dict (server reads only).
- M-TC3 server text-condition resolver (_reset). Reads cache; injects
  prompt_embeds; no co-resident swap when cache covers the prompt.
- M-TC4 deployment serial residency. Owner of: T5 / lingbot-va GPU
  residency lifecycle in the serve loop.

44.4 INTERFACE (as implemented)
- cache key: sha1(raw_prompt + "|" + str(max_seq)). Raw prompt (NOT
  prompt_clean) keeps the cache module free of any tokenizer/diffusers
  dependency; precompute + server pass the same raw string. max_seq=512
  (matches server._reset's encode_prompt(max_sequence_length=512)).
  negative_prompt "" cached under its own key.
- TextCondEntry (dataclass): prompt, max_sequence_length,
    prompt_embeds: Tensor[seq_len, dim], seq_len: int, dim: int.
- M-TC2 (text_cond_cache.py):
    cache_key(prompt, max_seq) -> str
    load_cache(path) -> dict[str, TextCondEntry]
    store_cache(path, entries) -> None
- M-TC1 CLI (precompute_text_cond.py): --videos_root (prompts read from
    obs_data 'task' fields under visualization/real/), --model_path,
    --output, --limit (smoke). stdout: per-prompt {seq_len, dim, l2}.
- M-TC3 (server): __init__ loads job_config.text_cond_cache (path) when
    set. _reset, on a (prompt, "") cache hit, injects the cached embeds
    to GPU inline and skips the T5 swap; miss / no cache -> the Phase-41
    v3 transient swap (retained as the robustness path). Residency is
    gated by cache presence, not a serve_residency enum.
- M-TC4 (deployment serial, 44d as implemented): serve_residency flag
    (job_config, default off). On a miss, _reset offloads the transformer
    to CPU, loads T5, encodes, offloads T5, reloads the transformer; the
    embeds land in an in-memory cache (_mem_text_cond) so only the first
    reset per unique prompt pays the swap. The encode runs at the top of
    _reset, BEFORE create_empty_cache allocates the KV buffer, so the
    encode window holds T5 alone: peak = max(T5, lingbot-va), not sum.
    _resolve_cached_embeds(prompt) is the single resolver for both the
    in-memory (44d) and disk (44c) caches.

44.5 FLOW
- precompute: enumerate prompts -> T5 on GPU -> encode each + neg ""
  -> store_cache -> free T5 -> exit.
- eval reset (precompute mode): _reset -> _resolve_cached_embeds(prompt)
  -> cache hit -> inject cached embeds -> T5 never loaded -> text_encoder
  stage peak = activation only. Coverage caveat (44.0 F1/F2): under
  demo_randomized the per-episode prompt is not reproducible, so hits
  require the eval to reuse the precompute source's exact prompts; a miss
  falls through to the 44d serial path (serve_residency on) or the 41 v3
  swap (off). For production demo_randomized the reporting path (44.0 F4
  per-stage diffusion peak) is the recommended apples-to-apples number.
- deployment serial: task start (lingbot-va not resident) -> load T5
  -> encode task prompts -> free T5 + empty_cache -> load lingbot-va
  -> run episodes -> free lingbot-va -> next task. Peak = max(T5,
  lingbot-va).
